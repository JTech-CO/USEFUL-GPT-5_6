# USEFUL GPT-5.6

> GPT-5.6을 일반 대화, 조사, 문서 작성, 파일 작업, 프로그래밍 및 Codex 작업에 효과적으로 활용하기 위한 한국어 프롬프트 가이드입니다.  
> OpenAI의 공식 **Prompting** 문서를 바탕으로 핵심 원칙을 실전형으로 재구성했습니다.

- 기준 문서: https://learn.chatgpt.com/docs/prompting
- 문서 언어: 한국어
- 최종 확인일: 2026-07-11
- 적용 대상: ChatGPT, ChatGPT Work, Codex IDE/CLI/Cloud
- 핵심 원칙: **결과를 먼저 명시하고, 필요한 맥락과 출력 형식, 중요한 경계조건만 추가합니다.**

# PART I. 소개

## 1. 프롬프트는 명령어 문법이 아니라 작업 명세입니다

프롬프트는 GPT-5.6에게 무엇을 **알아내고**, **만들고**, **수정하고**, **검증할지** 전달하는 작업 명세입니다. 특별한 구문이나 고정된 공식은 필요하지 않습니다.

간단한 질문은 짧게 작성해도 충분합니다. 다만 결과의 품질이나 정확성이 중요한 작업이라면 아래 네 요소를 포함하는 편이 좋습니다.

| 요소 | 의미 | 핵심 질문 |
|---|---|---|
| 목표(Goal) | 수행해야 할 최종 작업 | 무엇을 완성해야 하는가? |
| 맥락(Context) | 결과를 바꾸는 정보와 자료 | 어떤 배경·파일·출처를 참고해야 하는가? |
| 출력(Output) | 결과물의 형태와 수준 | 어떤 형식·분량·구조로 내놓아야 하는가? |
| 경계조건(Boundaries) | 변경 금지·금지 행동·승인 조건 | 무엇을 유지하고 무엇을 피해야 하는가? |

모든 항목을 억지로 채울 필요는 없습니다. 결과를 실질적으로 바꾸는 정보만 넣습니다.

## 2. 가장 유용한 기본 구조

```text
[목표]
무엇을 완성해야 하는지 한 문장으로 설명합니다.

[맥락]
대상, 독자, 현재 상황, 참고 자료, 데이터 범위를 제공합니다.

[출력]
형식, 분량, 구조, 언어, 상세 수준을 지정합니다.

[경계조건]
변경 금지 사항, 예산, 보안, 승인 필요 행동, 추측 금지 조건을 적습니다.

[검증]
완료 전에 무엇을 다시 확인하고 보고해야 하는지 지정합니다.
```

### 최소형

```text
첨부한 회의 메모를 프로젝트 팀이 바로 읽을 수 있는 짧은 진행 상황 보고서로 정리합니다.
결정 사항과 다음 행동을 먼저 배치하고, 담당자와 기한이 없는 항목은 별도로 표시합니다.
```

### 완성형

```text
월요일 경영진 회의용 1페이지 프로젝트 현황 보고서를 작성합니다.

최신 프로젝트 계획서와 관련 의사결정 기록을 사용합니다.
진행 상황, 주요 위험, 담당자, 기한, 경영진이 결정해야 할 사항을 정리합니다.

결정 요청과 다음 행동을 문서의 앞부분에 배치합니다.
승인된 일정과 예산 수치는 변경하지 않습니다.
자료가 충돌하거나 빠진 경우 추측하지 말고 명시합니다.
외부에 전송하거나 게시하지 말고 초안만 작성합니다.

완료 전에 모든 다음 행동에 담당자와 기한이 있는지 확인합니다.
```

## 3. 결과부터 설명합니다

세부 절차를 길게 지정하기 전에 **완성해야 할 결과물**을 먼저 명시합니다.

### 권장

```text
이 회의록을 개발팀용 주간 업데이트로 바꿉니다.
결정 사항, 차단 요소, 다음 행동 순서로 구성합니다.
```

### 비권장

```text
먼저 문장을 읽고, 핵심어를 찾고, 내용을 분류한 다음,
각 문단을 분석하고, 다시 요약해서 문서로 작성합니다.
```

작업 절차 자체가 중요하지 않다면 GPT-5.6이 자료를 탐색하고 비교하며 접근 방법을 조정할 여지를 남기는 편이 낫습니다.

절차를 명시해야 하는 경우는 다음과 같습니다.

- 법적·감사·품질보증 절차를 준수해야 할 때
- 재현 가능한 실험이나 테스트 순서가 필요할 때
- 파일을 변경하기 전에 계획 승인이 필요할 때
- 배포·전송·삭제처럼 되돌리기 어려운 행동이 포함될 때

---

## 4. 결과를 바꾸는 맥락만 제공합니다

맥락이 많다고 항상 정확도가 높아지는 것은 아닙니다. 결과를 실제로 바꾸는 정보만 제공합니다.

### 유용한 맥락

- 결과물을 읽을 대상
- 현재 문제와 원하는 최종 상태
- 반드시 사용해야 할 파일·데이터·출처
- 기간, 지역, 버전, 예산, 기술 스택
- 이미 확정된 결정
- 실패 증상과 재현 절차
- 참고해야 할 기존 스타일이나 코드 관례

### 자료 유형별 사용법

#### 문서·스프레드시트·프레젠테이션·PDF

```text
첨부한 세 개의 보고서를 비교해 공통 결론과 충돌하는 수치를 구분합니다.
모든 수치는 원본 파일과 연결해 표시하고, 확인할 수 없는 값은 추정하지 않습니다.
```

#### 이미지·스크린샷·다이어그램

이미지만 첨부하지 말고 어느 부분이 중요한지 설명합니다.

```text
첨부한 대시보드 스크린샷을 기준으로 새 화면을 구현합니다.
특히 상단 KPI 카드의 간격, 좌측 내비게이션 폭, 모바일에서의 카드 재배치를 재현합니다.
스크린샷에 보이지 않는 호버·키보드·검증 동작은 아래 요구사항을 따릅니다.
```

#### 웹 검색

최신 정보가 필요할 때 명시합니다.

```text
2026년 7월 현재 정보를 웹에서 확인해 비교합니다.
공식 문서와 1차 자료를 우선하고, 핵심 주장마다 출처를 표시합니다.
발행일과 실제 사건 발생일이 다르면 둘을 구분합니다.
```

#### 연결된 소스와 플러그인

검색 명령을 세세하게 작성하기보다 **어디에서 무엇을 찾아야 하는지** 지정합니다.

```text
Google Drive의 최신 프로젝트 계획서와 관련 Slack 채널의 최근 의사결정을 사용해 현황 보고서를 작성합니다.
```

```text
GitHub 저장소에서 관련 이슈, 최근 PR, 실패한 CI 기록을 확인해 원인을 정리합니다.
```

> 연결된 소스와 플러그인의 가용성은 사용 중인 요금제, 워크스페이스 설정 및 권한에 따라 달라질 수 있습니다.

## 5. 전역 선호와 작업별 지시를 분리합니다

여러 대화에서 반복되는 선호는 ChatGPT의 개인화 또는 사용자 지정 지침에 둡니다.

### 전역 설정에 적합한 항목

- 기본 언어
- 기본 문체
- 선호하는 설명 수준
- 표·코드·문서의 기본 스타일
- 반복적으로 사용하는 기술 스택
- 답변에서 지켜야 할 일반적인 표현 규칙

### 현재 프롬프트에 넣어야 하는 항목

- 이번 작업의 파일
- 이번 문서의 독자
- 특정 분량
- 이번 작업만의 예산과 일정
- 현재 버그의 재현 절차
- 일회성 금지 조건

## 6. 경계조건은 적고 강하게 작성합니다

경계조건은 결과가 쓸모없어지거나 의도하지 않은 행동이 발생하는 것을 막기 위한 지시입니다.

### 좋은 경계조건

```text
승인된 날짜와 예산 수치는 변경하지 않습니다.
```

```text
제공된 자료만 사용합니다. 빠진 정보는 추측하지 말고 표시합니다.
```

```text
총비용은 100만 원을 넘지 않도록 합니다.
```

```text
이메일은 초안만 작성하며 실제로 전송하지 않습니다.
```

```text
공개 API의 요청·응답 형식은 변경하지 않습니다.
```

### 좋지 않은 경계조건

- 모든 문장을 통제하려는 수십 개의 사소한 규칙
- 서로 충돌하는 지시
- 결과에 영향을 주지 않는 형식 강박
- “완벽하게”, “최대한”, “알아서 잘”처럼 검증할 수 없는 표현

중요한 경계조건 한두 개를 정확히 지정하는 것이 작은 규칙을 수십 개 나열하는 것보다 효과적입니다.

## 7. 결과물을 바로 사용할 수 있게 만듭니다

GPT-5.6에게 결과를 어디에 사용할지 알려주면 분량, 상세 수준, 구조를 더 정확히 선택할 수 있습니다.

```text
회의 직전 임원이 2분 안에 읽을 수 있는 1페이지 요약으로 작성합니다.
결정 요청과 다음 행동을 가장 먼저 배치합니다.
```

```text
이 메모를 고객에게 보낼 후속 이메일 초안으로 바꿉니다.
결정 사항, 담당자, 완료 기한을 포함합니다.
```

```text
계획 대비 실제 지출을 표로 만들고 차이가 10%를 넘는 항목을 강조합니다.
```

중요한 작업에서는 마지막 검증을 요청합니다.

```text
완료 전에 모든 행동 항목에 담당자와 기한이 있는지 확인합니다.
```

```text
확인하지 못한 정보와 사용한 가정을 별도 섹션에 정리합니다.
```

```text
본문과 첨부 표의 수치가 일치하는지 교차 검증합니다.
```

## 8. 첫 프롬프트를 완벽하게 만들 필요는 없습니다

첫 결과를 확인한 뒤 구체적인 수정 요청으로 조정합니다.

### 좋은 후속 요청

```text
도입부를 더 직접적으로 바꾸되 근거는 유지합니다.
권고안을 배경 설명보다 앞으로 옮깁니다.
```

```text
현재 구조는 유지하고 기술적 세부 수준만 한 단계 높입니다.
```

```text
표의 열은 유지하되 중복 행을 제거하고 위험도 순으로 정렬합니다.
```

```text
수정 범위를 헤더 컴포넌트로 제한합니다.
모바일 레이아웃에는 영향을 주지 않습니다.
```

### 후속 요청에서 유용한 표현

- “현재 구조는 유지하고 …만 변경합니다.”
- “이전 답변의 2번 항목만 다시 작성합니다.”
- “근거는 유지하되 분량을 절반으로 줄입니다.”
- “선택지 3개를 제시한 뒤 각각의 트레이드오프를 설명합니다.”
- “내가 수정한 부분을 덮어쓰지 않습니다.”

작업 중 방향을 바꾸는 **Steer**와 다음 작업으로 넘기는 **Queue**의 제공 여부와 조작 방법은 사용하는 ChatGPT 또는 Codex 환경에 따라 다를 수 있습니다.

# PART II. 일반 Chat에서의 프롬프트

## 9. 개념 설명

```text
복리의 원리를 투자 경험이 없는 사람에게 설명합니다.
구체적인 숫자 예시 하나를 사용하고, 처음 등장하는 금융 용어는 즉시 정의합니다.
마지막에 단리와의 차이를 3문장으로 요약합니다.
```

## 10. 글 작성 및 교정

```text
아래 초안을 전문적인 한국어 안내문으로 교정합니다.

독자: 기존 고객
목적: 서비스 점검 일정 안내
분량: 500자 이내
문체: 존댓말, 과장 없는 사실 전달
유지: 날짜, 시간, 문의처
금지: 새로운 보상 정책이나 확정되지 않은 내용 추가

완료 후 원문에서 의미가 불분명했던 부분만 별도로 표시합니다.
```

## 11. 선택지 비교

```text
아래 두 요금제를 연 2회 해외여행을 하는 1인 사용자 기준으로 비교합니다.
월 요금, 해외 데이터, 로밍 비용, 약정, 해지 비용을 표로 정리합니다.

그다음 하나를 추천하되, 추천이 뒤집히는 조건도 설명합니다.
확인할 수 없는 비용은 추정하지 않습니다.
```

## 12. 실행 계획

```text
평일 저녁 5일 식단을 계획합니다.

조건:
- 각 식사는 30분 이내
- 땅콩 제외
- 재료를 여러 식사에서 재사용
- 2인 기준
- 마지막에 하나의 통합 장보기 목록 제공

출력:
1. 요일별 메뉴
2. 예상 조리 시간
3. 재사용되는 재료
4. 통합 장보기 목록
```

# PART III. 복합 작업과 ChatGPT Work

## 13. Work가 적합한 작업

다음 유형은 단순 Chat보다 여러 자료와 도구를 사용하는 복합 작업에 적합합니다.

- 여러 출처를 읽고 비교하는 조사
- 순차적인 단계가 필요한 작업
- 파일 생성 또는 여러 산출물 작성
- 도구를 사용해 실제 변경을 수행하는 작업
- 반복 가능한 업무 흐름
- 결과물 간 일관성 검사가 필요한 작업

복합 작업에서는 아래 내용을 명확히 합니다.

1. 최종 결과물
2. 사용할 원본 자료
3. 결과물을 읽을 대상
4. 필수 작업과 선택적 개선의 구분
5. 검토 방법
6. 전송·게시·수정 전에 필요한 승인
7. 완료 전 검증 항목

## 14. 복합 작업 기본 템플릿

```text
[최종 결과]
완성해야 할 파일 또는 의사결정 자료를 설명합니다.

[자료]
사용해야 할 파일, 데이터, 연결된 소스, 검색 범위를 지정합니다.

[대상]
누가 읽고 어떤 결정을 내리는지 설명합니다.

[필수 작업]
```

## 18. Codex 프롬프트의 핵심 구조

코드 작업에서는 아래 다섯 가지가 중요합니다.

| 요소 | 설명 |
|---|---|
| 원하는 동작 | 무엇이 어떻게 작동해야 하는가 |
| 코드 범위 | 저장소, 파일, 함수, 경로, 관련 모듈 |
| 재현 절차 | 버그가 발생하는 정확한 순서 |
| 보존 조건 | API, 동작, 데이터 형식, 호환성 등 변경 금지 사항 |
| 검증 방법 | 실행할 테스트, 린트, 빌드, 수동 확인 절차 |

```text
[작업]
원하는 코드 변경을 설명합니다.

[범위]
관련 파일, 함수, 경로, 모듈을 지정합니다.

[재현 또는 근거]
버그 재현 절차나 요구사항의 출처를 제공합니다.

[제약]
공개 API, 호환성, 성능, 보안, 수정 범위를 지정합니다.

[검증]
실행할 테스트와 성공 조건을 적습니다.
```

멀티스텝 작업에서는 편집 전에 조사와 계획을 요청합니다. 사용하는 환경에서 지원되는 경우 `/plan`으로 접근 방식을 먼저 만들고, 지속 목표 기능이 제공될 때 `/goal`을 사용할 수 있습니다.

> 명령어와 기능은 Codex IDE, CLI, Cloud 버전에 따라 달라질 수 있으므로 현재 환경의 명령 목록을 확인합니다.

```text
- API 요청·응답 형식은 변경하지 않음

- 다른 설정 항목의 저장 동작은 변경하지 않음

```

```text

- 타이포그래피를 더 편집 디자인처럼 조정

- 나머지 섹션은 변경하지 않음
```

다음 반복:

```text
레이아웃은 유지합니다.
색상 수와 중복 테두리만 줄여 시각적 노이즈를 낮춥니다.
```

브라우저에서 직접 확인하고, 마음에 드는 변경은 커밋하며, 되돌린 변경이 있다면 다음 프롬프트에서 명시합니다.

# PART VII. 완료 전 체크리스트

## 40. 사용자 체크리스트

프롬프트를 보내기 전에 확인합니다.

- [ ] 최종 결과가 한 문장으로 명확한가?
- [ ] 대상 독자 또는 사용 목적이 필요한가?
- [ ] 결과를 바꾸는 자료와 맥락을 제공했는가?
- [ ] 출력 형식과 분량이 필요한 수준으로 지정되었는가?
- [ ] 가장 중요한 경계조건이 포함되었는가?
- [ ] 최신 정보가 필요하면 검색과 출처를 요청했는가?
- [ ] 외부 전송·게시·삭제·수정 전에 승인이 필요한가?
- [ ] 성공 여부를 판정할 검증 기준이 있는가?

## 41. 모델 완료 체크리스트

중요한 작업의 끝에 아래 검증을 요청할 수 있습니다.

```text
완료 전에 다음을 점검합니다.

- 모든 필수 요구사항이 반영되었는가
- 수치와 날짜가 출처 또는 입력과 일치하는가
- 사실, 분석, 가정이 구분되었는가
- 확인하지 못한 정보를 단정하지 않았는가
- 출력 형식과 분량이 요구사항을 충족하는가
- 코드라면 관련 테스트, 린트, 빌드를 실행했는가
- 파일이 여러 개라면 서로 모순되지 않는가
- 외부 행동이 승인 범위를 넘지 않았는가

문제가 있으면 수정한 뒤, 남은 불확실성만 보고합니다.
```

# 42. 핵심 요약

GPT-5.6에 유용한 프롬프트는 길어서 좋은 것이 아니라 **작업의 성공 조건이 선명해서** 좋습니다.

1. **결과를 먼저 말합니다.**
2. **결과를 바꾸는 맥락만 제공합니다.**
3. **형식과 사용 목적을 지정합니다.**
4. **중요한 경계조건만 강하게 설정합니다.**
5. **완료 전 검증 기준을 둡니다.**
6. **첫 결과를 보고 작은 후속 요청으로 조정합니다.**
7. **코드 작업에서는 재현 절차와 테스트 결과를 요구합니다.**
8. **외부 전송·게시·삭제는 승인 범위를 명확히 합니다.**

가장 간단한 형태로 압축하면 다음과 같습니다.

```text
{무엇을 완성할지} 수행합니다.
{결과를 바꾸는 자료와 맥락}을 사용합니다.
{형식, 분량, 대상}에 맞춰 작성합니다.
{변경 금지와 금지 행동}을 지킵니다.
완료 전에 {성공 조건}을 검증하고 확인하지 못한 부분을 표시합니다.
```
